# data-accuracy

## 目的

检查前端与后端的数据读写一致性，验证边界条件和异常输入处理，确保功能接口覆盖率达到 100%。

## 输入

- 前端代码（`index.html`）
- `API接口文档.md`
- `需求规格说明.md`

## 输出

- 数据准确性测试报告

## 执行步骤

### 1. 功能接口覆盖矩阵

对照需求文档中的所有功能点，建立覆盖矩阵：

```markdown
| 功能点 | 前端实现 | API端点 | 测试状态 |
|--------|----------|---------|----------|
| 用户注册 | registerForm() | POST /users | ✅ |
| 用户登录 | loginForm() | POST /auth/login | ✅ |
| 商品列表 | loadProducts() | GET /products | ✅ |
| ... | ... | ... | ... |

覆盖率 = 已测试功能点数 / 需求总功能点数 × 100%
目标：≥ 95%
```

### 2. 数据一致性测试

#### 2.1 写入-读取一致性

```
测试流程：
1. POST 创建资源 → 记录返回的 ID
2. GET 读取该资源 → 验证数据与写入时一致
3. PATCH 更新资源 → GET 验证更新生效
4. DELETE 删除资源 → GET 验证 404
```

#### 2.2 数据完整性

```markdown
| 测试项 | 操作 | 验证点 | 结果 |
|--------|------|--------|------|
| 必填字段 | 提交缺少必填字段的数据 | 返回 400 + 明确错误提示 | ✅ |
| 字段类型 | 字符串字段传入数字 | 返回 400 或自动类型转换 | ✅ |
| 字符串长度 | 用户名 200 字符 | 返回 400 (超过 VARCHAR(50)) | ✅ |
| 特殊字符 | 输入 SQL 注入语句 | 被转义/参数化，不报错 | ✅ |
| XSS | 输入 <script>alert(1)</script> | 输出时转义 | ✅ |
| Unicode | 输入 emoji 和中文 | 正确存储和返回 | ✅ |
| 并发写入 | 同时创建同名资源 | 一个成功，一个返回 409 | ❌ |
| 金额精度 | 99.999 → 存储和返回 | 四舍五入到 2 位小数 | ✅ |
| 时间戳 | 验证时区处理 | 统一 UTC / 带时区 | ✅ |
```

### 3. 边界条件测试

```markdown
| 测试项 | 输入 | 预期 | 结果 |
|--------|------|------|------|
| 空字符串 | name="" | 根据定义：NOT NULL 则 400 | ✅ |
| 极长文本 | name = 10000 字符 | 截断或 400 | ✅ |
| 负数 | price = -1 | 400 (价格不能为负) | ✅ |
| 零值 | quantity = 0 | 根据业务规则处理 | ✅ |
| 超大页码 | page = 99999 | 返回空列表（非错误） | ✅ |
| 未来时间 | date = 2099-01-01 | 根据业务规则处理 | ✅ |
| NULL 值 | 必填字段传 null | 400 | ✅ |
```

### 4. 前端数据流测试

- [ ] 表单提交前是否正确校验（减少无效 API 请求）
- [ ] API 返回错误时前端是否正确展示错误信息
- [ ] 数据更新后页面是否重新渲染正确数据
- [ ] 筛选/排序后显示的数据是否正确
- [ ] 分页切换后数据是否正确加载
- [ ] 浏览器刷新后数据状态是否正确恢复
