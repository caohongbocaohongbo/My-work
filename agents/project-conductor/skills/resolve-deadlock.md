# resolve-deadlock

## 目的

当 agent 执行超过 25 分钟时，诊断卡住原因并尝试解决。

## 输入

- 卡住的 agent 名称
- 已用时间
- 当前中间输出

## 输出

- 诊断报告
- 解决方案（重试 / 拆分 / 跳过 / 人工介入）

## 执行步骤

1. **分析中间输出**：
   - 检查是否有死循环（重复输出相同内容）
   - 检查是否在等待外部资源（API 无响应、文件未生成）
   - 检查是否因上下文过长导致推理变慢
   - 检查是否因任务粒度过大导致单步耗时过长

2. **分类诊断**：

   | 症状 | 诊断 | 解决方案 |
   |------|------|----------|
   | 重复输出相同内容 > 5 次 | 死循环 | 强制重定向到下一步 |
   | 等待 API/网络响应 > 10 分钟 | 网络阻塞 | 跳过该调用，使用 mock 数据 |
   | 上下文 tokens > 100K | 上下文过载 | 拆分任务为更小粒度 |
   | 任务本身复杂度过高 | 粒度问题 | 拆分为 2-3 个子任务分别执行 |

3. **执行解决方案**：
   - 方案 A（重试）：重置 agent，用更精简的 prompt 重新执行
   - 方案 B（拆分）：将当前任务拆成更小的子任务，逐一执行
   - 方案 C（跳过）：标记该步骤为"需人工处理"，继续下一个 agent
   - 方案 D（求助）：暂停流程，生成详细错误报告，请求人工介入

4. **生成诊断报告**：
   ```
   [死锁诊断报告]
   - Agent: <名称>
   - 卡住原因: <分类>
   - 已尝试方案: <A/B/C/D>
   - 当前状态: <已解决 / 需人工介入>
   - 建议后续: <继续流程 / 暂停等待>
   ```
