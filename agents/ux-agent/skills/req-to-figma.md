# req-to-figma

## 目的

将需求文档转化为 Figma 中的设计方案，替代原来的 `设计稿.html` 产出。**每次执行至少提供 2 个方案**，设计完成后需经用户在 Figma 中手动对比确认，确认通过后输出带 node-id 的 Figma 链接给 front-agent 进行 figma-to-code。

## 前置条件

- req-agent 已通过 KPI 检查（需求覆盖度 >=95%）
- `需求规格说明.md` 已就绪
- **DESIGN.md 或 README.md 已确认存在**（见下方阻断规则），除非用户明确下指令跳过

### DESIGN.md 阻断规则

在执行任何设计操作之前：
1. 检查项目根目录是否存在 `DESIGN.md`，存在则完整读取
2. 若 `DESIGN.md` 不存在，查找 `README.md` 作为替代
3. 两者都不存在 → **立即阻断**，通知用户添加参考文档，不得继续执行
4. 例外：用户**明确说**"不需要参照 DESIGN.md"或"忽略 DESIGN.md"时可跳过

## 输入

- `需求规格说明.md`（由 req-agent 产出）
- 线框图布局方案（来自 `wireframe-pencil.md`，≥2 个）
- 视觉主题参数（来自 `style-theme-tune.md`，≥2 套）

## 输出

- Figma 设计文件链接（格式：`https://figma.com/design/<fileKey>/<fileName>?node-id=<nodeId>`）
- 每个方案独立 Page，含完整页面设计、组件、交互状态
- 方案差异说明
- 用户确认通过标记 + 选中方案的 node-id

## 方案要求

每次执行至少提供 **≥2 个可用方案**，常见维度：

| 方案维度 | 示例 | Figma 呈现方式 |
|----------|------|---------------|
| 配色方案 | 方案 A：科技蓝 #2563EB / 方案 B：深海蓝 #1E40AF | 不同 Page |
| 布局方案 | 方案 A：居中窄版 (960px) / 方案 B：全宽 (1200px) | 不同 Page |
| 组件风格 | 方案 A：圆角柔和 / 方案 B：直角硬朗 | 不同 Page |
| 层级结构 | 方案 A：单页滚动 / 方案 B：Tab 切换分区 | 不同 Page |
| 卡片样式 | 方案 A：白底轻阴影 / 方案 B：浅灰底无阴影 | 不同 Page |
| 主次关系 | 方案 A：左侧专业版突出 / 方案 B：中间专业版突出 | 不同 Page |

**方案命名规范**：`方案A-<差异描述>`、`方案B-<差异描述>`，如：
- `方案A-科技蓝配色`
- `方案B-深海蓝配色`

## 执行步骤

### 阶段 1：设计准备

1. **读取 DESIGN.md / README.md**（阻断点）：
   - 检查项目是否存在 `DESIGN.md`，不存在则查 `README.md`
   - 两者都没有 → 阻断并通知用户添加
   - 用户明确说"不需要参照"时可跳过
   - 提取样式约束、视觉规范、组件风格要求

2. **读取需求文档**，提取：
   - 完整页面/视图清单
   - 每个页面的功能元素和交互流程
   - 数据展示需求（表格、图表、卡片等）

3. **运行 `wireframe-pencil.md`**：
   - 产出 ≥2 个页面结构树和布局方案
   - 标注各方案的差异点和适用场景
   - 确定每个页面的区域划分

4. **运行 `style-theme-tune.md`**：
   - 产出 ≥2 套配色方案和视觉主题
   - 每套包含完整的 CSS 变量体系
   - 标注各方案的风格关键词和适用行业

### 阶段 2：Figma 设计执行

4. **创建 Figma 设计文件**：
   - 使用 `mcp__figma-hosted__create_new_file` 创建新 Figma 设计文件
   - 文件命名规范：`<项目名称> - UX设计稿`

5. **为每个方案创建独立 Page**：
   - Page 命名：`方案A-<描述>`、`方案B-<描述>`
   - 在每个 Page 内按该方案的布局+主题搭建完整设计

6. **逐方案逐页构建设计**（使用 `use_figma`）：
   - 创建 Frame（对应桌面/平板/手机断点）
   - 按照该方案的线框图摆放组件和内容
   - 应用该方案的视觉主题参数
   - 设置 Auto Layout

7. **实现核心交互状态**（每个方案）：
   - 按钮：default / hover / active / disabled / loading
   - 输入框：default / focus / error / disabled / filled
   - 卡片：default / hover / selected
   - 弹窗/抽屉：打开/关闭状态
   - 导航：当前页高亮、展开/收起
   - 空态、加载态、错误态页面

8. **填充模拟数据**：
   - 使用逼真的中文 mock 数据
   - 表格数据 5-10 行
   - 列表数据覆盖正常态和空态

### 阶段 3：用户确认

9. **呈现方案给用户选择**：
   - 提供 Figma 文件链接
   - **逐方案说明差异点和特点**，格式如下：
     ```markdown
     ## 设计方案对比
     
     ### 方案 A：<方案名称>
     - 配色：<描述>
     - 布局：<描述>
     - 特点：<核心优势>
     - Figma Page：`方案A-<描述>`
     
     ### 方案 B：<方案名称>
     - 配色：<描述>
     - 布局：<描述>
     - 特点：<核心优势>
     - Figma Page：`方案B-<描述>`
     
     ### 差异对比表
     | 维度 | 方案 A | 方案 B |
     |------|--------|--------|
     | 配色主调 | ... | ... |
     | 布局宽度 | ... | ... |
     | 卡片风格 | ... | ... |
     ```

10. **用户在 Figma 中手动确认**：
    - 用户在 Figma 中浏览各方案 Page
    - 用户选择其中一个方案，或提出具体修改意见
    - 如果用户要求修改：在 Figma 对应 Page 中直接改
    - 循环迭代直到用户明确确认通过（最多 3 次）

11. **用户确认通过后**：
    - 标记选中方案为「已确认」
    - 删除未被选中的方案 Page（保持文件整洁）
    - 提取选中方案的 node-id
    - 组装完整 Figma 链接

### 阶段 4：交接给 front-agent

12. **输出交接信息**：
    ```markdown
    ## UX 设计交接

    ### 用户确认的方案
    - 方案名称：<最终选中方案>
    - Figma 链接: https://figma.com/design/<fileKey>/<fileName>?node-id=<nodeId>
    - fileKey: <fileKey>
    - nodeId: <nodeId>
    - 确认状态: 用户已确认

    ### 设计概要
    - 页面数: N 个
    - 组件数: N 个
    - 视觉主题: <主题描述>
    - 断点: 桌面 / 平板 / 手机

    ### 被淘汰的方案
    - 方案 X：<简述淘汰原因>

    ### 备注
    - <特殊说明或注意事项>
    ```

## Figma MCP 工具使用指引

### 创建新文件
```
mcp__figma-hosted__create_new_file
  fileName: "<项目名> - UX设计稿"
  planKey: "<用户 plan key>"
  editorType: "design"
```

### 在 Figma 中创建方案 Page
使用 `use_figma` 编写 Plugin API 脚本：
- 创建多个 Page，命名为 `方案A-<描述>`、`方案B-<描述>`
- 每页内创建 Frame 和 Auto Layout
- 创建组件并设置 variant 属性
- 绑定颜色/字体变量
- 填充文本和模拟数据

### 获取设计 context（供 front-agent 后续使用）
```
mcp__figma-hosted__get_design_context
  fileKey: "<fileKey>"
  nodeId: "<nodeId>"
```

## 自检清单

- [ ] ≥2 个方案已创建，每个方案独立 Page
- [ ] 所有需求页面均已在每个方案中覆盖
- [ ] 核心交互状态已覆盖
- [ ] 颜色/字体/间距方案间有明显差异可对比
- [ ] 方案差异说明已整理
- [ ] 用户在 Figma 中已确认选中方案
- [ ] 未选中方案已清理
- [ ] node-id 已正确提取
- [ ] 交接信息完整无误

## 异常处理

- **无法创建 Figma 文件**：检查 planKey 和权限，回退到 Pencil `.pen` 文件方案
- **用户 3 次反馈仍不满意**：暂停流程，生成差异报告，请求人工介入
- **Figma API 超时**：拆分设计任务为多个小批次执行
- **node-id 提取失败**：使用 `get_metadata` 确认页面结构后重新提取
- **方案数量不足 2 个**：必须在执行前规划至少 2 个有意义的差异方案，不能仅做表面差异
