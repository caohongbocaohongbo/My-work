# regression-check

## 目的

在 front-agent 或 back-agent 修复问题后，重新测试核心用例，确保修复没有引入新问题（回归）。

## 输入

- 上一轮测试报告（含失败用例清单）
- 修复后的代码/API
- `需求规格说明.md`（核心功能列表）

## 输出

- 回归测试报告

## 执行步骤

### 1. 确定回归范围

```markdown
回归测试范围：
├── 第一优先级：上一轮失败的用例（必须全部通过）
├── 第二优先级：与修复模块相关的核心功能
└── 第三优先级：关键业务流程（端到端）
```

### 2. 回归测试套件

#### 第一优先级：失败用例复测

直接复现上一轮失败的用例：
```markdown
| 原用例ID | 描述 | 上一轮结果 | 本轮结果 | 状态 |
|----------|------|------------|----------|------|
| TC-03 | pageSize 超限 | ❌ 500 | ✅ 400 | 已修复 |
| TC-12 | 并发写入冲突 | ❌ 无处理 | ✅ 409 | 已修复 |
```

#### 第二优先级：关联模块冒烟测试

修复模块可能影响的相关功能：
```markdown
修复模块: 用户参数校验中间件

关联模块冒烟:
- ✅ GET /users 列表正常分页
- ✅ POST /users 正常创建
- ✅ GET /users/:id 正常获取
- ✅ PATCH /users/:id 正常更新
- ✅ DELETE /users/:id 正常删除
```

#### 第三优先级：核心业务流程端到端

```markdown
流程: 用户注册 → 登录 → 浏览商品 → 下单

步骤:
1. ✅ POST /users (注册) → 201
2. ✅ POST /auth/login (登录) → 200 + token
3. ✅ GET /products (商品列表) → 200
4. ✅ POST /orders (创建订单) → 201
5. ✅ GET /orders/:id (查看订单) → 200
```

### 3. 回归判定标准

| 结果 | 判定 | 动作 |
|------|------|------|
| 失败用例全部通过 + 关联模块 100% 通过 | ✅ 回归通过 | 流程继续 |
| 失败用例全部通过 + 关联模块有失败 | ⚠️ 新问题 | 通知对应 agent 修复 |
| 失败用例仍有未通过 | ❌ 未修复 | 通知对应 agent 重新修复 |

### 4. 输出格式

```markdown
## 回归测试报告

### 修复验证
- 上轮失败用例数: 5
- 本轮通过: 4
- 本轮仍失败: 1
- 修复成功率: 80%

### 关联模块冒烟测试
- 测试模块数: 3
- 通过: 3
- 失败: 0

### 核心流程端到端
- 测试流程数: 2
- 通过: 2
- 失败: 0

### 新发现的问题
[如有新问题，详细描述]

### 结论
- [ ] 全部通过，可进入下一阶段
- [ ] 存在未修复问题，需继续修复
- [ ] 发现新问题，需通知对应 agent
```
