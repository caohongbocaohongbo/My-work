# 设计审查报告生成（Design Review Report）

编排 design-spec-compliance 和 component-consistency-check 的审查结果，计算综合合规度，按标准模板生成审查报告，判定 PASS/FAIL。

## 执行流程

### 1. 聚合审查数据

从两个审查技能提取：

**风格一致性**（权重 40%）：
- 颜色检查项 N，通过 N → 颜色得分 = 通过/总数 × 100%
- 字体检查项 N，通过 N → 字体得分
- 间距检查项 N，通过 N → 间距得分
- 形状检查项 N，通过 N → 形状得分
- 风格一致性总分 = 各子项等权平均

**交互统一性**（权重 30%）：
- 组件状态覆盖检查项 N，通过 N
- 全局状态覆盖检查项 N，通过 N
- 点击区域检查项 N，通过 N
- 交互一致性总分 = 通过数/总检查项数 × 100%

**组件统一性**（权重 30%）：
- 组件使用率 × 0.7 + 属性正确率 × 0.3

### 2. 计算综合合规度

```
综合合规度 = 风格×0.4 + 交互×0.3 + 组件×0.3
```

保留一位小数。

### 3. 违规分级

| 级别 | 定义 | 扣分权重 | 示例 |
|------|------|----------|------|
| 🔴 高危 | 与 DESIGN.md 完全不一致 | ×3 | 主色不同、核心组件缺失、字体家族不一致 |
| 🟡 中危 | 部分不一致 | ×1 | 间距偏差 > 4px，组件属性未设置 |
| 🟢 低危 | 微小偏差 | ×0.5 | 行高偏差 2px，圆角偏差 1px |

### 4. 判定逻辑

```
IF 综合合规度 >= 99% AND 高危违规数 == 0 THEN
  PASS → "放行给 front-agent"
ELSE
  FAIL → "退回 ux-agent 返工，修复高危+中危违规后重新提交"
```

**硬性门禁**：即使综合合规度 >= 99%，存在高危违规仍判 FAIL。

### 5. 输出审查报告

按 README.md 模板输出 8 章节完整报告，写入 `design-review/审查报告.md`。

向 project-conductor 返回结构化摘要：

```
判定: PASS/FAIL
综合合规度: XX.X%
风格一致性: XX% | 交互统一性: XX% | 组件统一性: XX%
违规: 🔴N 项 🟡N 项 🟢N 项
报告路径: design-review/审查报告.md
```

若 PASS → conductor 将 Figma 链接 + 审查报告交付 front-agent
若 FAIL → conductor 将审查报告退回 ux-agent，附带修复建议

## 返工循环

- 同一设计稿最多 2 次返工
- 第 3 次审查仍不通过 → 暂停流程，生成升级报告，请求人工介入
- 每次审查必须重新获取 Figma 数据（不得复用缓存）

## 关键规则

- **修复建议必须可执行**：指明"将 node-id X 的 fill 从 #AAA 改为 #BBB"
- **审查报告必须完整输出**：无缺省章节
- **不通过 ≠ 失败**：是正常质量控制流程
