# 组件一致性审查（Component Consistency Check）

检查 Figma 设计稿是否使用了设计系统组件，检测自定义替代，确保组件级别一致性。

## 执行流程

### 1. 获取设计系统组件清单

从 DESIGN.md 提取 Figma 组件参考链接：
- 关键词：`组件库`、`Component Library`、`Design System`、`figma.com/design/`
- 有链接 → 调用 `get_metadata` 获取组件库结构，列出所有 Component/ComponentSet
- 无链接 → 调用 `search_design_system` 或 `get_libraries` 检索关联设计库

输出组件清单：
| 组件名称 | 类型 | 关键属性 | 用途 |
|----------|------|----------|------|
| Button/Primary | ComponentSet | size, state, icon | 主操作按钮 |
| ... | ... | ... | ... |

### 2. 遍历设计稿节点树

用 `get_metadata` 获取完整节点树，区分：
- **组件实例（INSTANCE）**：使用了某组件的实例
- **自定义节点（FRAME/GROUP/RECTANGLE/TEXT）**：基础图形拼凑的"伪组件"

### 3. 组件使用对照

对每个设计系统组件，在设计稿中检查：

1. **是否被使用**：搜索对应 INSTANCE 节点；关键组件完全缺失 → 标记"全部用自定义替代"
2. **属性是否正确**：variant 是否选对、暴露属性是否填写、是否有手动 override 组件子节点
3. **是否存在自定义替代**：本应使用组件却用 FRAME+RECTANGLE+TEXT 拼凑 → 严重违规
4. **尺寸是否正确**：是否选择了正确的 size variant（非手动拉伸）

### 4. 自定义组件检测

判断每个非实例 UI 元素：

- 装饰性元素（分割线/背景色块）→ 允许
- 容器布局（Auto Layout Frame）→ 允许
- **本应使用设计系统组件** → 违规，标记"自定义替代"

判定标准（任一满足即判定）：
- 视觉外观与某设计系统组件高度相似
- 功能角色（按钮/输入框/卡片）在设计系统中有对应组件
- 需要该组件的状态管理能力

### 5. 组件使用率计算

```
组件使用率 = 设计系统组件实例数 / (组件实例数 + 自定义替代数) × 100%
```

达标：>= 95%。装饰性和布局容器不计入分母。

### 6. 额外检查

- **Detached Instance**：脱离主组件的实例 → 违规
- **图标**：设计系统有图标库时，使用图标组件而非外部 SVG/Image
- **组件 Override 过度**：同时 override 颜色+字体+间距 → 失去组件一致性优势 → 半违规

## 输出格式

向 `design-review-report.md` 提供：

```markdown
## 组件审查结果

| 设计系统组件 | 使用数 | 状态 |
|-------------|--------|------|
| Button/Primary | 5 | ✅ |
| Navbar | 0 | ❌ 用自定义 Frame 替代 |
| ...

### 违规清单
| # | Node ID | 描述 | 应使用 | 实际 | 修复建议 |
|---|---------|------|--------|------|----------|

### 统计
- 组件实例数：N
- 自定义替代数：N
- 组件使用率：XX.X%（阈值 >= 95%）
```
