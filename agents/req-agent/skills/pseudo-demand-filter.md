# pseudo-demand-filter

## 目的

识别并剔除伪需求、过度设计和非必要项，保持需求清单精简有效。

## 输入

- `requirement-extract.md` 产出的需求点清单

## 输出

- 过滤后的需求清单（标记了被剔除项及原因）

## 伪需求识别规则

1. **无用户受益**：实现了但用户不会感知或用不到的功能。
2. **过度抽象**："系统应支持所有可能的支付方式" → 改为具体支持的支付方式列表。
3. **技术实现细节伪装**："使用 Redis 缓存用户 session" → 应改为"用户登录状态应在服务重启后保持"。
4. **脱离核心场景**：与项目主要目标无关的边缘 case。
5. **矛盾需求**：两个需求逻辑上无法同时满足。
6. **无限扩展性**："系统应支持无限用户" → 应改为具体数字或合理上限。
7. **过早优化**："数据库应使用分库分表架构"（在 MVP 阶段）。

## 执行步骤

1. **逐条审查**：对每个需求点，问三个问题：
   - 这个需求解决什么问题？
   - 不实现它会有什么影响？
   - 有没有更简单的替代方案？
2. **标记分类**：
   - ✅ 保留：确实是核心需求
   - ❌ 剔除：伪需求/过度设计
   - ⚠️ 降级：从 P0 降为 P1/P2
   - 🔄 改写：保留意图，但换一种更合理的表述
3. **输出过滤报告**：
   ```markdown
   ## 需求过滤报告
   ### 保留的需求（共 X 条）
   ...
   ### 剔除的需求（共 Y 条）
   - [原需求] → 剔除原因：[具体原因]
   ### 降级的需求（共 Z 条）
   - [原需求] P0 → P2：降级原因
   ### 改写的需求（共 W 条）
   - [原需求] → [新表述]
   ```
